# Prompt-only model bake-off: Gemma 3 4B vs Mistral 7B vs Qwen3-8B

Date: 2026-07-09
Machine: Mac mini, Apple M4, 16 GB unified memory

## Purpose and scope

HLD SS13.5's model portfolio table names Mistral 7B Instruct v0.3 and
"Qwen3.5-9B" as measured challengers for the primary model, "never promoted
merely because it is faster" and with "no runtime dependency." This is that
bake-off: the same 24-case corpus used in
`aeroroute-mlx/docs/QUALITY_CORPUS_2026-07-09.md`, run against all three
models under the identical prompt template and validators, one model per
process for clean peak-memory isolation.

This is a prompt-only comparison (config A in HLD SS13.9's A/B/C scheme --
no few-shot examples, no QLoRA adapter for any of the three). It is not a
promotion decision; the primary model (Gemma 3 4B, the validated fallback
per `aeroroute-mlx-training/docs/COMPATIBILITY_12B_2026-07-09.md`) was never
a candidate to begin with here.

## Model substitution: "Qwen3.5-9B" does not exist as a text-only model

HLD names "Qwen3.5-9B" as the challenger. The exact HuggingFace identifiers
matching that name (`mlx-community/Qwen3.5-9B-MLX-4bit`,
`mlx-community/Qwen3.5-9B-4bit`) are **vision-language models**, loaded via
`mlx_vlm`, not `mlx_lm` -- confirmed by fetching both model cards. HLD SS13.5
explicitly scopes this project to text-only architectures ("vision inputs
and mlx-vlm are outside this project," stated for Gemma 3 and applied here
by the same logic). Loading either would require a different runtime
dependency (`mlx-vlm`) and image-capable prompt plumbing this project does
not have and is not scoped to build for a text-explanation task.

Substituted `mlx-community/Qwen3-8B-4bit` (confirmed text-only,
`pipeline_tag: text-generation`, loads via `mlx_lm`, Apache-2.0): same
Qwen model family, closest available text-only parameter count (8B vs the
named 9B), no code changes needed since it uses the same `mlx_lm` API as
the other two models.

## A real environment bug found and fixed along the way

A fresh `uv sync --extra mlx` in this repo resolved `transformers==5.13.0`,
which crashes on import with `AttributeError: 'str' object has no attribute
'__module__'` inside `AutoTokenizer.register` (mlx-lm registers a custom
tokenizer class at import time; something about how 5.13.0 changed
`auto_factory.register`'s type-checking breaks that call). `aeroroute-mlx`'s
existing lock file happened to predate 5.13.0's release, so it was silently
protected by timing rather than an explicit pin. Both `aeroroute-mlx` and
`aeroroute-mlx-training`'s `pyproject.toml` now pin `transformers<5.13.0` in
the `mlx` extra so a future `uv lock --upgrade` doesn't reintroduce this.

## Results

| Model | Pass rate | Latency p50 | Latency p95 | Peak RSS |
| --- | ---: | ---: | ---: | ---: |
| **Gemma 3 4B** (baseline) | **100%** (23/23) | **2.595 s** | **2.973 s** | 3,369.8 MB |
| Mistral 7B Instruct v0.3 | 100% (23/23) | 3.889 s | 5.253 s | 3,207.8 MB |
| Qwen3-8B | **43.5%** (10/23) | 9.145 s | 9.335 s | 3,774.5 MB |

(23 evaluated per model; the same one case, `OMDB-SKBO-B77W-minimum_time`,
is excluded for all three -- it is a correct `aircraft_mass_outside_profile`
API rejection unrelated to the model, see
`aeroroute-mlx/docs/QUALITY_CORPUS_2026-07-09.md`.)

All three fit comfortably in 16 GB with headroom -- memory is not a
differentiator among these three.

### Gemma 3 4B vs Mistral 7B

Both hit 100% schema/validation pass rate. Gemma 3 4B is meaningfully
faster (roughly 1.8x at p95). Nothing in this corpus favors Mistral over
the existing baseline.

### Qwen3-8B

13 of 23 cases failed with `model output is not a JSON object` -- the
strict-JSON prompt contract (`Return exactly one JSON object with this
schema: {"text":"..."}`) was not reliably honored. Qwen3 models are known
to default to an extended "thinking" generation mode that prepends
reasoning content before the final answer; the ~9s latency (3-3.5x slower
than the other two models on an identical prompt and token budget) is
consistent with that explanation, though the raw pre-parse model output
was not captured by this run's error handling to confirm directly (a gap
worth fixing before relying on this script again). Whatever the exact
mechanism, Qwen3-8B did not reliably follow this project's structured-
output contract with the current prompt and default chat template.

## Decision

**No promotion.** Gemma 3 4B remains the model used by
`aeroroute-mlx`. Per HLD SS13.5, a challenger is "never promoted merely
because it is faster" -- neither challenger is even that: Mistral matches
quality at higher latency, and Qwen3-8B fails the structured-output
contract on more than half the corpus outright. This closes the challenger
bake-off obligation from the HLD model portfolio table with a real,
reproducible result rather than leaving it unmeasured.

## What would change this

- Qwen3-8B could plausibly be made usable by explicitly disabling its
  thinking mode (many Qwen3 chat templates expose an `enable_thinking`
  flag or `/no_think` convention) and re-running -- not attempted here, out
  of scope for this bake-off pass.
- If a future task requires vision input (none currently planned; HLD
  explicitly excludes it), the real "Qwen3.5-9B" vision-language model
  would need its own ADR, `mlx-vlm` dependency, and image-capable prompt
  design -- a materially different scope than this text-explanation
  service.

## Reproduce

```bash
cd aeroroute-mlx-training
uv sync --all-groups --extra mlx
find .venv -name '._*' -delete
uv run python scripts/model_bakeoff.py \
  --repo-id mlx-community/Mistral-7B-Instruct-v0.3-4bit \
  --revision a4b8f870474b0eb527f466a03fbc187830d271f5 \
  --local-path models/mistral-7b-instruct-v0.3-4bit \
  --label mistral-7b-instruct-v0.3 \
  --output docs/bakeoff-mistral-7b-2026-07-09.json
# repeat with --repo-id mlx-community/Qwen3-8B-4bit, revision
# 545dc4251c05440727734bcd94334791f6ab0192, --local-path models/qwen3-8b-4bit
```

(Requires the local dev stack up, same as `quality_corpus.py`.)
